# Error envelope

Non-2xx responses use:

```json
{
  "ok": false,
  "error": {
    "code": "INVALID_JSON_BODY",
    "message": "Invalid JSON body",
    "detailed": {
      "message": "Expected a valid E.164 phone number in \"to\""
    }
  }
}
```

## Actual status-code behavior

| Status | What the backend currently uses it for |
| --- | --- |
| `400` | Validation failures, invalid content type, unsupported phone numbers, and route-specific bad input |
| `401` | Missing or invalid API key |
| `403` | Missing permission, subscription required, billing block, or other auth/business guard failures |
| `404` | Unknown route or unknown resource id |
| `429` | Rate limited |
| `500` | Internal or provider-side failure surfaced by the backend |

The current backend does not use `422` for request validation. Validation errors are returned as `400 INVALID_JSON_BODY`.

## Rate-limit headers

On `429`, the backend sets:

- `X-RateLimit-Global`
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Retry-After`
- `X-RateLimit-Retry-Policy`

Important details:

- `X-RateLimit-Retry-After` is returned in milliseconds.
- The error body also includes `error.detailed.retryAfterInSeconds`.
- The current backend does not add the standard `Retry-After` header.

Example `429` payload:

```json
{
  "ok": false,
  "error": {
    "code": "RATE_LIMITED",
    "message": "Too many requests. Try again later",
    "detailed": {
      "policy": "50;w=60;scope=\"project\"",
      "scope": "project",
      "limit": 50,
      "route": "POST /messages",
      "retryAfterInSeconds": 60
    }
  }
}
```

## Retry guidance

Retry only when failure is likely transient:

- `429`
- transient `5xx`

Do not blindly retry:

- `400 INVALID_JSON_BODY`
- `401 INVALID_OR_MISSING_API_KEY`
- `403 MISSING_PERMISSION`
- `403 PROJECT_SUBSCRIPTION_REQUIRED`
- `403 PROJECT_BILLING_PAYMENT_REQUIRED`

## Idempotency

`POST /messages`, `POST /messages/batch`, and `POST /otp` accept `Idempotency-Key`.

Current backend behavior:

- Reusing the same key for the same logical request replays the stored result.
- The backend does not currently surface idempotency conflicts as `409`.
- OTP payload mismatches on a reused key return `400 INVALID_JSON_BODY`.

Example:

```bash
curl -X POST "https://api.rewritetoday.com/v1/messages" \
  -H "Authorization: Bearer rw_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: msg:login-otp:2026-03-19T18:30" \
  -d '{
    "to": "+5511999999999",
    "content": "Rewrite: your verification code is 123456"
  }'
```
