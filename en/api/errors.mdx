# Error envelope

Non-2xx responses use:

```json
{
  "ok": false,
  "error": {
    "code": "INVALID_JSON_BODY",
    "message": "Invalid JSON body",
    "detailed": {
      "message": "Expected a valid E.164 phone number in \"to\""
    }
  }
}
```

## Actual status-code behavior

| Status | What the backend currently uses it for |
| --- | --- |
| `400` | Validation failures, invalid content type, unsupported phones, sandbox scheduling attempts, and idempotency payload mismatches |
| `401` | Missing or invalid API key |
| `403` | Missing permission, subscription required, billing guard, webhook limits, or API key mode mismatch |
| `404` | Unknown route or unknown resource ID |
| `409` | Conflict-style guards such as duplicated webhook endpoints |
| `429` | Rate limited |
| `500` | Internal failure or provider-facing backend error |

## Rate-limit headers

In `429`, the backend sends:

- `X-RateLimit-Global`
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Retry-After`
- `X-RateLimit-Retry-Policy`

Important details:

- `X-RateLimit-Retry-After` is returned in milliseconds.
- The error body also includes `error.detailed.retryAfterInSeconds`.
- The backend does not currently add the standard `Retry-After` header.

## Retry guidance

Retry only when the failure is likely transient:

- `429`
- transient `5xx`

Do not blindly retry:

- `400 INVALID_JSON_BODY`
- `401 INVALID_OR_MISSING_API_KEY`
- `403 MISSING_PERMISSION`
- `403 PROJECT_SUBSCRIPTION_REQUIRED`
- `403 PROJECT_BILLING_PAYMENT_REQUIRED`
- `403 API_KEY_MODE_MISMATCH`

## Idempotency

`POST /messages`, `POST /messages/batch`, and `POST /otp` accept `Idempotency-Key`.

Current backend behavior:

- Reusing the same key for the same logical operation reuses the stored result.
- OTP idempotency mismatches return `400 INVALID_JSON_BODY`.
- The public API does not currently expose a dedicated idempotency conflict response.
