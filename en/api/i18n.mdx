# Overview

Rewrite supports an `i18n` key for both direct messages and templates.

Use `message` as your default fallback, then provide localized variants in `i18n`.
Rewrite resolves the best variant based on the destination number (`to`) country/region inferred from E.164.

## When to use i18n

- You send the same flow to multiple countries.
- You want one API integration with locale-specific copy.
- You need safe fallback behavior when no locale match exists.

## Payload shape

```json
{
  "to": "+5511999999999",
  "message": "Your OTP code is 123456",
  "i18n": {
    "pt-BR": "Seu codigo OTP e 123456",
    "es": "Tu codigo OTP es 123456",
    "en": "Your OTP code is 123456"
  }
}
```

## Resolution order

1. Rewrite inspects the destination number (`to`) prefix and infers locale/country.
2. If an exact locale match exists in `i18n` (for example `pt-BR`), Rewrite uses it.
3. If no exact locale exists, Rewrite may use a language-level key when available (for example `pt`).
4. If no mapping is found, Rewrite uses `message`.

<Info>
Always send a complete `message` fallback. Do not rely on `i18n` keys only.
</Info>

## Supported i18n key formats

- Language: `en`, `pt`, `es`
- Language + region: `en-US`, `pt-BR`, `es-MX`

BCP-47 style keys are recommended for predictable matching.

## Templates + i18n

When using templates, `i18n` follows the same behavior:

- Keep a default template/message.
- Add locale-specific variants in `i18n`.
- Keep variables/placeholders consistent across all locales.

## Best practices

- Keep placeholders identical across locales (example: `{{code}}`).
- Keep message length under regional carrier constraints.
- Prefer explicit locales (`pt-BR`) when tone/wording changes by country.
- Add monitoring by locale to spot missing mappings.
- Test with real phone formats from your top countries.

## Common mistakes

<Tabs sync={false}>
  <Tab title="No fallback message">
    Sending only `i18n` keys without a default `message` can break delivery for unmatched destinations.
  </Tab>
  <Tab title="Inconsistent placeholders">
    If one locale uses different placeholder names, runtime interpolation can fail.
  </Tab>
  <Tab title="Wrong locale key format">
    Avoid custom keys like `brazilian-portuguese`. Use BCP-47 style keys instead.
  </Tab>
</Tabs>
