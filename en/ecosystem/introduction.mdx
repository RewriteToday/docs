The Rewrite ecosystem is a set of focused packages designed to work together.
Use them independently or combine them for a typed, resilient integration stack.

## Ecosystem architecture

<CardGroup cols={2}>
  <Card title="Adapters" icon="plug" color="#C9CBD1">
    Framework-native and runtime-native webhook entrypoints with signature verification and typed events.
  </Card>

  <Card title="Contracts" icon="code" color="#C9CBD1">
    Shared enums, route builders, and payload contracts for consistent integrations.
  </Card>

  <Card title="Validation" icon="shield-check" color="#C9CBD1">

  <Card title="Transport" icon="route" color="#C9CBD1">

  <Card title="Product layer" icon="box" href="/en/sdks/introduction" color="#C9CBD1">

## Which piece should I use?

<Accordion title="I need to receive Rewrite webhooks in my framework">
  Start with an adapter. Adapters are the fastest way to mount a verified, typed webhook endpoint inside Express, Fastify, Elysia, Hono, or a standard edge runtime.
</Accordion>

<Accordion title="I want fast integration with less boilerplate">
  Start with the official SDKs. They provide higher-level APIs and are usually the fastest path for product teams.
</Accordion>

<Accordion title="I need full HTTP/runtime control">
  Use `@rewritetoday/rest`, optionally combined with `@rewritetoday/types` for typed routes and payloads.
</Accordion>

<Accordion title="I only need contracts and route builders">
  Use `@rewritetoday/types` by itself when you already have your own HTTP client or framework abstraction.
</Accordion>

<Accordion title="I need runtime validation for external input or webhooks">
  Use `@rewritetoday/zod` or `@rewritetoday/typebox` to parse payloads at the boundary before your business logic runs, depending on your validation stack.
</Accordion>

<Accordion title="I need webhook docs and event semantics">
  Use the Webhooks docs together with an adapter page. The Webhooks section explains delivery format, signatures, retry behavior, and event families.
</Accordion>
