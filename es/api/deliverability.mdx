# Resumen

La entregabilidad es la probabilidad de que un mensaje sea aceptado por los operadores y llegue al dispositivo del destinatario en una ventana de tiempo util.

En Rewrite, la entregabilidad se trata como un ciclo continuo entre calidad de envio, calidad de ruta y feedback de eventos.

## Como funciona la entrega en Rewrite

1. Validacion y normalizacion de la solicitud
2. Decision de proveedor y ruta
3. Despacho y traspaso al operador
4. Ingestion de feedback por webhook
5. Actualizaciones de politica de la aplicacion (supresion, reintento, escalado)

## Pilares de entregabilidad

<Tabs sync={false}>
  <Tab title="Higiene de listas">
    - Envia solo a numeros E.164 válidos.
    - Elimina destinos con fallos duros repetidos.
    - Suprime de inmediato a usuarios dados de baja.
    - Evita listas obsoletas o con muchos numeros reciclados.
  </Tab>
  <Tab title="Calidad de contenido">
    - Mantén textos claros y esperados.
    - Identifica la marca al inicio del mensaje.
    - Evita patrones tipo spam (enlaces engañosos, todo en mayusculas, abuso de puntuacion).
    - Mantén mensajes transaccionales y OTP cortos y precisos.
  </Tab>
  <Tab title="Comportamiento de envio">
    - Evita picos bruscos de volumen en rutas nuevas.
    - Incrementa gradualmente por pais/caso de uso.
    - Reintenta solo fallos transitorios con backoff + jitter.
    - Respeta ventanas horarias del usuario.
  </Tab>
  <Tab title="Cumplimiento">
    - Conserva evidencia de consentimiento cuando sea obligatorio.
    - Incluye instrucciones de baja cuando la normativa o caso de uso lo exija.
    - Separa tráfico transaccional y promocional.
    - Sigue restricciones regionales para reglas de remitente/plantilla.
  </Tab>
</Tabs>

## Reducir resultados negativos

| Resultado | Causa tipica | Accion |
| --- | --- | --- |
| `undelivered` | Filtrado del operador, terminal inalcanzable, estrés de ruta | Mejora seguridad del contenido, valida calidad del destino, reintenta segun politica |
| `failed` | Destino inválido, rechazo de politica, error duro aguas abajo | Clasifica como permanente/transitorio, suprime permanentes |
| Entrega retrasada | Congestion, presión de ruta | Usa reintentos acotados y monitorizacion de SLO de latencia |
| Picos de quejas/bajas | Tráfico irrelevante o frecuencia alta | Endurece segmentacion y controles de frecuencia |

## Politica de reintento para entregabilidad

Reintenta solo cuando el fallo probablemente sea transitorio:

- `429` y `5xx` transitorios
- timeout temporal del proveedor
- saturacion temporal de ruta

No reintentes de forma ciega fallos permanentes de validacion/politica.

Base de implementacion:
- clave de idempotencia por envio lógico
- backoff exponencial + jitter
- limite maximo de intentos
- cola de dead-letter para inspeccion manual

Implementacion de referencia: [Errores y reintentos](/es/api/errors)

## Estrategia de calentamiento de throughput

1. Empieza con volumen regional controlado.
2. Aumenta gradualmente mientras monitorizas entrega y quejas.
3. Mantén flujos separados para OTP y tráfico similar a promocional.
4. Pausa el ramp-up cuando empeoren los ratios de fallo.

## Bucle de optimizacion guiado por webhooks

1. Ingiere eventos de webhook casi en tiempo real.
2. Persiste payloads normalizados.
3. Actualiza estado de salud y supresion del destinatario.
4. Genera alertas al superar umbrales.
5. Ajusta ruta y politica para envios siguientes.

## Payload practico para envios listos para analitica

```json
{
  "to": "+15551234567",
  "message": "Rewrite: tu codigo de verificacion es 123456",
  "i18n": {
    "pt-BR": "Rewrite: seu codigo de verificacao e 123456"
  },
  "metadata": {
    "use_case": "otp_login",
    "tenant": "acme-app",
    "campaign": "auth-v1",
    "template_version": "v3"
  }
}
```

Usa `metadata` para segmentar la entregabilidad por region, caso de uso y version de plantilla.

## Checklist de entregabilidad

- Valida E.164 antes de enviar.
- Aplica reglas de consentimiento y supresion.
- Mantén textos claros, cortos e identificados por marca.
- Usa [i18n](/es/api/i18n) con fallback seguro.
- Reintenta solo fallos transitorios mediante [Errores y reintentos](/es/api/errors).
- Monitorea resultados por region/proveedor/caso de uso.
- Auto-suprime destinos con fallos permanentes.
- Revisa semanalmente la taxonomia de fallos y ajusta politicas.
