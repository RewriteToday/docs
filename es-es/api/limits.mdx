## Modos de facturacion

<Tabs sync={false}>
  <Tab title="PAYG">
    PAYG se factura por cada SMS enviado.

    **Precio:** `0.20` por SMS (segun la politica comercial de Rewrite).

    Este modelo es ideal para tráfico variable o cargas en fase inicial.
  </Tab>
  <Tab title="Planes">

    Los niveles actuales, volumen incluido y comportamiento de sobreuso se publican en:
    [rewritetoday.com/pricing](https://rewritetoday.com/pricing)
  </Tab>
</Tabs>

## Dimensiones de cuota

| Dimension | Que controla | Señal tipica |
| --- | --- | --- |
| Tasa de solicitudes API | Rafagas/throughput hacia endpoints API | `429` + headers de reintento |
| Volumen de envio SMS | Capacidad total de envio segun facturacion | Contadores de uso en dashboard |
| Ventana de cuota del plan | Volumen incluido en ciclo del plan | Estado de uso del plan |

## Que pasa cuando alcanzas limites

- La presion de API puede devolver `429` con hints de reintento.
- Los nuevos envios pueden bloquearse al agotarse la cuota comercial.
- El comportamiento depende del modo de facturacion y politica del workspace (PAYG vs plan).

Guia relacionada: [Errores y reintentos](/es-es/api/errors)

## Headers de rate limit a revisar

Cuando recibas rate limit, revisa:

- `X-RateLimit-Global`
- `X-RateLimit-Limit`
- `X-RateLimit-Retry-After`
- `Retry-After`

Si existen varios hints de reintento, usa el tiempo de espera mas conservador.

## Recomendaciones practicas de politica

1. Define limites blandos internos antes de limites duros de cuota.
2. Alerta cuando el uso llegue al 70%, 85% y 95% de la ventana.
3. Reserva presupuesto para casos de uso de alta prioridad (ej.: OTP).
4. Auto-limita campañas de menor prioridad durante picos de presion.
5. Mantén colas dead-letter para envios que superen limites de politica.

## Checklist de monitorizacion

- Monitorea tasa de `429` por endpoint y entorno.
- Monitorea ratios de envio exitoso, failed y undelivered por region.
- Monitorea coste por caso de uso (`otp_login`, `alerts`, `marketing`).
- Concilia contadores internos con resultados de proveedor/webhooks.
- Revisa incidencias de cuota semanalmente y ajusta politicas de throttling.
