# Vue d'ensemble

Les limites d'utilisation définissent comment le trafic est admis, facturé et protégé dans Rewrite.

Cette page explique :

- Le fonctionnement des quotas de requêtes et d'envoi
- Le modèle de facturation PAYG
- Les quotas par plan et les upgrades
- Ce qu'il se passe aux frontières de limite
- Les bonnes pratiques opérationnelles

## Modes de facturation

<Tabs sync={false}>
  <Tab title="PAYG">
    PAYG est facturé par SMS envoyé.

    **Prix :** `0.20` par SMS (selon la politique commerciale Rewrite).

    Ce modèle est optimal pour un trafic variable ou des charges de travail en phase initiale.
  </Tab>
  <Tab title="Plans">

## Dimensions des quotas

| Dimension | Ce que cela contrôle | Signal typique |
| --- | --- | --- |
| Débit de requêtes API | Rafale/débit vers les endpoints API | `429` + en-têtes de retry |
| Volume d'envoi SMS | Capacité totale d'envoi selon le mode de facturation | Compteurs d'usage du tableau de bord |
| Fenêtre de quota de plan | Volume inclus sur le cycle du plan | Statut d'usage du plan |

## Que se passe-t-il quand vous atteignez les limites

- La pression API peut renvoyer `429` avec des indications de retry.
- Les nouveaux envois peuvent être bloqués une fois le quota commercial épuisé.
- Le comportement dépend du mode de facturation et de la politique de l'espace de travail (PAYG vs plan).

Guide associé : [Erreurs et tentatives](/fr/api/errors)

## En-têtes de rate limit à lire

En cas de rate limit, inspectez :

- `X-RateLimit-Global`
- `X-RateLimit-Limit`
- `X-RateLimit-Retry-After`
- `Retry-After`

Si plusieurs indications de retry sont présentes, utilisez le délai d'attente le plus conservateur.

## Recommandations pratiques de politique

1. Définissez des limites internes souples avant les limites dures de quota.
2. Alertez quand l'usage atteint 70 %, 85 % et 95 % de la fenêtre.
3. Réservez du budget pour les cas d'usage prioritaires (ex. OTP).
4. Auto-limitez les campagnes de moindre priorité en cas de pression.
5. Gardez des files dead-letter pour les envois qui dépassent les limites de politique.

## Checklist de monitoring

- Suivez le taux de `429` par endpoint et environnement.
- Suivez les ratios d'envoi réussi, failed et undelivered par région.
- Suivez le coût par cas d'usage (`otp_login`, `alerts`, `marketing`).
- Réconciliez les compteurs internes avec les résultats fournisseur/webhook.
- Révisez les incidents de quota chaque semaine et ajustez la politique de throttling.
