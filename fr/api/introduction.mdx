## Base URL

Utilisez cet host pour tous les endpoints API :

```text
https://api.rewritetoday.com
```

Exemple d'endpoint de creation de message :

```text
https://api.rewritetoday.com/v1/messages
```

<Info>
Utilisez uniquement HTTPS. N'envoyez jamais de cles API sur des canaux non securises.
</Info>

## Versioning

Rewrite versionne les endpoints dans le path URL.

Version documentee actuelle :

- `v1`

Regles de versioning :

- Les evolutions backward-compatible peuvent sortir dans la meme version.
- Les changements breaking sortent sous un nouveau path de version.
- Gardez la version explicite dans tous les endpoints hardcodes.

## Authentification

Authentifiez chaque requete avec une API key secrete depuis votre dashboard Rewrite.

Header attendu :

```http
Authorization: Bearer rw_...
```

Exemple de requete :

```bash
curl -X POST "https://api.rewritetoday.com/v1/messages" \
  -H "Authorization: Bearer rw_..." \
  -H "Content-Type: application/json" \
  -d '{
    "to": "+15551234567",
    "message": "Your OTP code is 123456"
  }'
```

<Warning>
N'exposez jamais des API keys secretes dans du code client web/mobile. Gardez-les cote serveur.
</Warning>

## Conventions API

- `Content-Type` : utilisez `application/json` pour les payloads JSON.
- Format numero : utilisez E.164 (exemple : `+15551234567`).
- Erreurs : les reponses non-2xx renvoient des payloads d'erreur structures.
- Limites : une forte pression de requetes peut renvoyer `429` avec des headers de retry.
