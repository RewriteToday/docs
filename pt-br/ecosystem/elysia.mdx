[`@rewritetoday/elysia`](https://www.npmjs.com/package/@rewritetoday/elysia) e o adapter da Rewrite para apps Elysia que querem uma rota de webhook nativa com pouca cerimonia.

Repositorio: [github.com/RewriteToday/adapters](https://github.com/RewriteToday/adapters)

## Quando usar

- Quando seu backend ja usa roteamento e handlers do Elysia.
- Quando quer um adapter orientado ao framework em vez de montar a verificacao manualmente.
- Quando prefere callbacks tipados a codigo customizado de parse de webhook.
- Quando quer uma fronteira clara, no nivel da rota, para eventos de entrada da Rewrite.

## Exemplo basico

```ts
import { Elysia } from 'elysia';
import { Webhooks } from '@rewritetoday/elysia';

const app = new Elysia();

app.post(
  '/webhooks/rewrite',
  Webhooks({
    secret: process.env.REWRITE_WEBHOOK_SECRET!,
    onPayload({ id, type, data }) {
      console.log({ id, type, data });
    },
  }),
);
```

## Boas praticas

<Accordion title="Mantenha o segredo apenas no servidor">
  Carregue o segredo de assinatura a partir de variaveis de ambiente e mantenha esse valor separado de qualquer configuracao exposta ao cliente.
</Accordion>

<Accordion title="Trate a rota como fronteira de entrada">
  Sempre que possivel, deixe a logica de negocio fora do callback da rota para manter verificacao, logging e processamento posteriores mais simples.
</Accordion>

<Accordion title="Deduplique entregas em retry">
  Webhooks da Rewrite podem ser reenviados. Armazene o id do evento e torne os efeitos colaterais idempotentes.
</Accordion>

<Accordion title="Callbacks de evento disponiveis">
  O adapter suporta `onSMSOTP`, `onMessageSent`, `onMessageBatch`, `onMessageQueued`, `onMessageDelivered`, `onMessageScheduled`, `onMessageFailed`, `onMessageCanceled` e o hook generico `onPayload`.
</Accordion>
