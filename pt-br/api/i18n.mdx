# Visao geral

A criacao publica de mensagens nao aceita um campo `i18n` direto no body.

A localizacao acontece hoje na camada de template:

- `POST /templates` e `PATCH /templates/{id}` aceitam um objeto `i18n`.
- `GET /templates` e `GET /templates/{identifier}` podem incluir `i18n` com `withi18n=true`.
- `POST /messages` resolve o conteudo localizado a partir de `templateId` e do pais inferido em `to` ou no telefone do contato salvo.

## Formato das chaves `i18n`

As chaves do mapa usam codigos ISO de pais em lowercase, por exemplo:

- `br`
- `us`
- `mx`

A maior parte dos projetos hoje usa trafego `br`, mas a resolucao de locale segue o pais final do destino quando o projeto tem numeros internacionais habilitados.

## Crie um template localizado

```bash
curl -X POST "https://api.rewritetoday.com/v1/templates" \
  -H "Authorization: Bearer rw_..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "otp_login",
    "content": "Rewrite: seu codigo e {{code}}",
    "i18n": {
      "br": "Rewrite: seu codigo e {{code}}"
    },
    "variables": [
      { "name": "code", "fallback": "000000" }
    ],
    "description": "SMS de OTP para login"
  }'
```

## Envie com `templateId`

```bash
curl -X POST "https://api.rewritetoday.com/v1/messages" \
  -H "Authorization: Bearer rw_..." \
  -H "Content-Type: application/json" \
  -d '{
    "to": "+5511999999999",
    "templateId": "748395130237498600",
    "variables": {
      "code": "123456"
    }
  }'
```

## Consulte as variantes salvas

```bash
curl "https://api.rewritetoday.com/v1/templates/otp_login?withi18n=true" \
  -H "Authorization: Bearer rw_..."
```

## Notas

- `content` continua sendo o fallback padrao.
- `i18n` fica no template, nao no body de `POST /messages`.
- Mantenha os nomes das variaveis identicos em todas as variantes.
