# Visao geral

A criacao publica de mensagens nao aceita um campo `i18n` direto no body.

Hoje a localizacao aparece em dois pontos:

- `GET /templates` e `GET /templates/{identifier}` podem incluir variantes salvas quando voce envia `withi18n=true`.
- `POST /messages` resolve o conteudo do template a partir de `templateId`, do pais do destino e de quaisquer variantes localizadas ja existentes naquele template.

## Formato das chaves `i18n`

As chaves usam codigos de pais em lowercase, por exemplo:

- `br`
- `us`
- `mx`

## Consulte variantes salvas

```bash
curl "https://api.rewritetoday.com/v1/templates/otp_login?withi18n=true" \
  -H "Authorization: Bearer rw_..."
```

Exemplo de resposta:

```json
{
  "ok": true,
  "data": {
    "id": "748395130237498600",
    "name": "otp_login",
    "content": "Rewrite: your code is {{code}}",
    "i18n": {
      "br": "Rewrite: seu codigo e {{code}}"
    },
    "variables": [
      { "name": "code", "fallback": "000000" }
    ],
    "description": "SMS de OTP para login",
    "tags": {
      "flow": "otp"
    },
    "createdAt": "2026-05-06T18:30:00.000Z"
  }
}
```

## Notas

- `content` continua sendo o fallback padrao.
- As rotas publicas de create/update de template nao documentam mais escrita de `i18n`.
- Mantenha os nomes das variaveis identicos em todas as variantes salvas.
