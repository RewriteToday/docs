# Envelope de erro

Respostas nao-2xx usam:

```json
{
  "ok": false,
  "error": {
    "code": "INVALID_JSON_BODY",
    "message": "Invalid JSON body",
    "detailed": {
      "message": "Expected a valid E.164 phone number in \"to\""
    }
  }
}
```

## Comportamento real de status code

| Status | Como o backend usa hoje |
| --- | --- |
| `400` | Falhas de validacao, content type invalido, telefones nao suportados e erros de input da rota |
| `401` | API key ausente ou invalida |
| `403` | Permissao ausente, assinatura obrigatoria, bloqueio de billing ou outros guards de negocio |
| `404` | Rota desconhecida ou id inexistente |
| `429` | Rate limited |
| `500` | Falha interna ou erro de provider propagado pelo backend |

O backend atual nao usa `422` para validacao. Erros de validacao voltam como `400 INVALID_JSON_BODY`.

## Headers de rate limit

Em `429`, o backend envia:

- `X-RateLimit-Global`
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`
- `X-RateLimit-Retry-After`
- `X-RateLimit-Retry-Policy`

Detalhes importantes:

- `X-RateLimit-Retry-After` vem em milissegundos.
- O body de erro tambem inclui `error.detailed.retryAfterInSeconds`.
- O backend atual nao adiciona o header padrao `Retry-After`.

Exemplo de payload `429`:

```json
{
  "ok": false,
  "error": {
    "code": "RATE_LIMITED",
    "message": "Too many requests. Try again later",
    "detailed": {
      "policy": "50;w=60;scope=\"project\"",
      "scope": "project",
      "limit": 50,
      "route": "POST /messages",
      "retryAfterInSeconds": 60
    }
  }
}
```

## Guia de retry

Retente apenas quando a falha for provavelmente transiente:

- `429`
- `5xx` transientes

Nao retente cegamente:

- `400 INVALID_JSON_BODY`
- `401 INVALID_OR_MISSING_API_KEY`
- `403 MISSING_PERMISSION`
- `403 PROJECT_SUBSCRIPTION_REQUIRED`
- `403 PROJECT_BILLING_PAYMENT_REQUIRED`

## Idempotencia

`POST /messages`, `POST /messages/batch` e `POST /otp` aceitam `Idempotency-Key`.

Comportamento atual do backend:

- Reusar a mesma key para a mesma operacao logica reaproveita o resultado salvo.
- O backend nao expoe hoje conflito de idempotencia como `409`.
- Mismatch de payload de OTP na mesma key retorna `400 INVALID_JSON_BODY`.

Exemplo:

```bash
curl -X POST "https://api.rewritetoday.com/v1/messages" \
  -H "Authorization: Bearer rw_..." \
  -H "Content-Type: application/json" \
  -H "Idempotency-Key: msg:login-otp:2026-03-19T18:30" \
  -d '{
    "to": "+5511999999999",
    "content": "Rewrite: seu codigo de verificacao e 123456"
  }'
```
